- **问题 2：** 登录/支付失败
- **问题 3：** 接口 404 / 500

## 9.5 阿里云 / Docker 部署说明

### 服务器基础信息

- **云厂商：** 阿里云
- **主机名：** `iZbp1edbm4ajavhqp98srsZ`
- **操作系统：** Ubuntu 24.04.3 LTS
- **内核：** 6.8.0-90-generic
- **Docker 版本：** 28.2.2
- **Docker Compose 版本：** 2.37.1

### 当前容器结构

| 容器名 | 镜像 | 作用 | 端口/暴露 |
|---|---|---|---|
| `nginx-prod` | `nginx:1.18` | 对外 Web 入口 / 反向代理 | 宿主机 `8080 -> 容器80` |
| `php-prod` | `shanghaicatjump-prod-php:latest` | PHP-FPM / ThinkPHP 应用运行容器 | 容器内 `9000/tcp` |
| `mysql-prod` | `mysql:8.0` | MySQL 数据库 | 容器内 `3306/tcp`、`33060/tcp` |

### 代码与配置挂载关系

- **Nginx 配置挂载：**
  - 宿主机：`/root/deploy/nginx/default.conf`
  - 容器内：`/etc/nginx/conf.d/default.conf`

- **Nginx 对外静态目录挂载：**
  - 宿主机：`/root/deploy/php/public`
  - 容器内：`/var/www/html/public`

- **PHP 项目代码挂载：**
  - 宿主机：`/root/deploy/php`
  - 容器内：`/var/www/html`

### 当前运行方式判断

- `nginx-prod` 启动命令：`nginx -g 'daemon off;'`
- `php-prod` 启动命令：`php-fpm`
- 当前结构为 **Nginx + PHP-FPM 分容器部署**
- Nginx 容器通过配置把 PHP 请求转给 `php:9000`
- 项目代码目前位于宿主机 `/root/deploy/php`
- 当前未在 `/root/deploy` 下发现标准 `docker-compose.yml` / `compose.yml`，需确认是否为手工 `docker run` 启动，或 compose 文件存放在其他目录

### 对外访问链路（当前推断）

- 外部请求先到 **宿主机 8080**
- 再进入容器 **nginx-prod:80**
- PHP 动态请求再转发到 **php-prod:9000**
- MySQL 由容器 `mysql-prod` 提供

> 备注：如果线上域名最终走的是 80/443，再由宿主机 Nginx / 安全组 / 面板转到 8080，需要补充宿主机层的转发规则。

## 9.7 小微接手后的标准工作流（SOP）

### 场景 A：收到线上异常 / Bug

1. 小微先收集信息：报错现象、影响范围、复现路径、发生时间
2. 优先查看相关日志：`/root/deploy/php/runtime`、`docker logs`、Nginx 配置与容器状态
3. 输出初步判断：是前端问题、后端问题、配置问题、入口层问题，还是第三方平台问题
4. 若只是排查结论和修复建议，小微可直接提交文档/说明
5. 若需要改代码，小微在仓库中提交 PR / 补丁，并给出测试建议
6. 若涉及重启、改配置、正式环境操作，先通知负责人并等待确认
7. 故障处理完成后，补充文档与复盘要点

### 场景 B：新增功能 / 需求开发

1. 负责人提出需求目标
2. 小微先评估：改哪些页面、接口、表结构、第三方依赖，风险在哪
3. 小微输出开发方案与影响范围
4. 小微完成代码修改并提交 PR / 补丁
5. 小微给出验证清单：功能点、回归点、风险点
6. 负责人验收后，手动上传体验版 / 提审 / 正式发布
7. 发布后，如有必要，小微补充 docs 与变更记录

### 场景 C：日常巡检

1. 查看核心服务是否在线：宿主机 Nginx、`nginx-prod`、`php-prod`、`mysql-prod`
2. 查看最近错误日志与异常波动
3. 检查域名 / HTTPS / 接口可用性
4. 检查是否有明显磁盘、日志、缓存积压问题
5. 输出巡检结论：正常 / 风险提示 / 建议处理项

### 场景 D：发布配合

1. 小微整理本次改动说明
2. 小微列出发布前检查项
3. 负责人手动完成微信小程序上传 / 提审 / 正式发布
4. 小微协助做发布后验证：首页、登录、下单、支付、订单、客服等关键链路
5. 若发现问题，小微先定位是代码、配置、入口层还是第三方依赖问题

### 默认协作原则
